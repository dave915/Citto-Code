# 에이전트 팀 가이드

## 언제 쓰면 좋은가

에이전트 팀은 답 하나를 빨리 받기 위한 기능보다, **여러 관점에서 같은 주제를 검토**할 때 가장 잘 맞습니다.

- 새 기능 설계 전에 빠진 요구사항을 찾고 싶을 때
- 출시 전에 버그, 보안, 테스트 누락을 같이 보고 싶을 때
- UX/UI 개선안을 제품, 디자인, 개발 관점으로 나눠 보고 싶을 때
- 성능 이슈를 구현, 병목, 회귀 위험까지 같이 점검하고 싶을 때

단순한 단답형 질문이나 파일 하나만 바로 고치면 되는 작업은 일반 채팅이 더 효율적입니다.

## 빠른 시작

1. 팀을 만들고 에이전트를 2명 이상 고릅니다.
2. 목적에 맞는 토론 모드를 선택합니다.
3. 열린 질문 형태로 주제를 입력합니다.
4. 토론이 끝나면 각 에이전트의 발언을 비교해서 결론을 정리합니다.

## 추천 팀 구성

### 기능 설계팀
설계자 + 개발자 + 비판자 + 테스터

- 언제: 새 기능 구조를 잡기 전
- 잘 맞는 질문:

```text
이 기능 설계를 검토해줘. 빠진 요구사항, 구현 리스크, 테스트 포인트를 각자 관점에서 정리해줘.
```

### 제품 개선팀
설계자 + UI 디자이너 + 개발자 + 비판자

- 언제: 온보딩, 전환율, 사용성 개선
- 잘 맞는 질문:

```text
온보딩 이탈률을 낮추는 방안을 제품, 디자인, 구현 관점에서 각각 제안해줘.
```

### 출시 점검팀
개발자 + 테스터 + 보안전문가 + 비판자

- 언제: 배포 직전 위험 확인
- 잘 맞는 질문:

```text
이 변경사항을 출시해도 되는지 검토해줘. 버그 가능성, 보안 위험, 테스트 누락을 중심으로 봐줘.
```

### 성능 최적화팀
개발자 + 최적화 전문가 + 비판자 + 테스터

- 언제: 성능 병목 분석, 회귀 위험 점검
- 잘 맞는 질문:

```text
이 화면이 느린 원인을 분석하고, 효과 대비 구현 비용이 좋은 최적화 순서로 제안해줘.
```

## 토론 모드

### 순차 (`→`)
앞 에이전트의 응답을 참고하면서 차례로 발언합니다.

- 깊이 있는 검토에 적합
- 앞선 의견에 동의, 반박, 보완이 자연스럽게 이어짐

### 병렬 (`⇉`)
모든 에이전트가 동시에 독립적으로 1차 응답을 만듭니다.

- 초기 응답이 서로에게 덜 끌려감
- 빠른 브레인스토밍에 적합

### 회의 (`◎`)
여러 라운드를 반복하며 의견을 좁혀 갑니다.

- 중요한 설계 결정이나 찬반이 갈리는 주제에 적합
- 시간이 더 걸리지만 합의 과정이 잘 보임

## 좋은 주제 쓰는 법

좋은 주제는 **판단 대상 + 제약 + 원하는 산출물**이 함께 들어갑니다.

좋은 예시:

```text
Electron 세션 상태 구조를 리팩터링할지 검토해줘. 기존 IPC 이벤트 이름은 유지해야 하고, 마이그레이션 비용이 낮아야 해. 각자 문제점, 대안 2개, 추천안까지 정리해줘.
```

피하는 게 좋은 예시:

```text
이 함수 이름 뭐가 좋을까?
```

## 결과 읽는 방법

- 캐릭터를 누르면 우측 상세 패널에서 해당 에이전트 전체 발언을 볼 수 있습니다.
- `Latest cue`를 누르면 해당 라운드 시작 지점으로 이동합니다.
- 말풍선 우측 상단의 복사 아이콘으로 개별 발언을 복사할 수 있습니다.
- 중앙 `TASK` 보드를 누르면 현재 주제를 전체 길이로 다시 확인할 수 있습니다.

## 조작 팁

- `Enter`: 전송
- `Shift + Enter`: 줄바꿈
- 스트리밍 중 `Esc`를 빠르게 두 번 누르면 토론을 중단할 수 있습니다.
- 팀은 최대 8명까지 구성할 수 있지만, 보통 4~6명이 가장 읽기 쉽습니다.
